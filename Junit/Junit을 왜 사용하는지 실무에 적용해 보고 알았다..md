## Junit을 왜 사용하는지 실무에 적용해 보고 알았다.

### 역시 백문이 불여일견
누가 말해주어 듣고 글로 읽고 하는 Input은 역시 내 머릿속에 확실히 각인 되기는 어려웠다.  
역시 잘 와닿지 않고 이해하고 싶을 때는 직접 해보고 느껴보는 것이다. 오늘은 그동안 TDD, Junit, Mock 왜 필요하고 무엇이 좋은지 알게 되어 글로 남겨 본다.


### 빠른 주기로 빠르게 테스트 할 수 있다.
기존에는 테스트 하기 위해서 WAS를 띄우고 Spring에 @PostConstruct를 사용해서 테스트 하였다. (DB 접속이 필요하기 때문이다.) 이것에 장점은 테스트 하기 위해 너무 느리고 또한 단위테스트 하나 하려는데 쓸데없이 WAS까지 띄우는게 굉장히 비효율적이다. 

하지만, Junit을 통해 테스트 DB 객체를 만들어 놓고 테스트하고 싶은 테스트 클래스에서 Junit만 실행시켜 주면 빠르게 테스트 하여 금방 내코드에 피드백을 받을 수 있어 기존 개발 방식보다 좀 더 품질이 좋은 코드를 만들 확률이 높아진다고 느껴졌다.  
또한 디버깅도 바로 할 수 있어 오류도 빨리 찾을 수 있다.  

그래도 와 닿지 않는가?? 

### WAS 띄우고 결과 확인까지 약 7초
단위테스트 하기위해 7초 기다리다가 결과 확인했는데 어이쿠 이런 오류 나서 다시 수정해야 한다.  다시 수정 후 다시 WAS 띄우고 7초..  
이렇게 고쳐질때까지 반복하는 경우와

### Junit으로 결과 확인까지 약 2~3초
바로 결과 확인후 수정하고 디버깅모드로 오류 빠르게 찾고 2~3초내 다시 피드백 받고 수정하고

어느것이 빠른가?? 이것이 내가 Junit을 사용한 결과 가장 편리함 점이 었다. **빠른 주기로 빠르게 테스트 할 수 있다는 점**

### 그래서 TDD?
거창하게 TDD를 FM으로 해보진 않아서 100%으로 이해까지는 무리고 있지만 취지나 테스트 중요성에 대해서는 알게됬다.  

실제 개발에 들어가기전에 작성할 로직을 Junit을 통해 빠르게 피드백 받고 검증된 로직을 개발할 클래스에 붙여 넣으면서 개발하는게 심적으로도 안정되고 오히려 개발 속도도 더 빠르게 느껴졌다.

### 앞으로 
Junit에서 제공하는 기능을 많이 적용해보고 익숙해져서 활용을 높이고 개발 품질과 속도를 더 높여봐야 겠다.
